## 二、冲突发现与修订

### 冲突 1：首页结构

**所在文档**：PRODUCT\_v0.2.md、PROTOTYPE\_v0.2.md、TECH\_SPEC\_v0.2.md

**原口径**：

- 首页为单一日期流矩阵，从今天开始向下延伸
- 筛选栏提供时间范围筛选（今天/近7天/近30天/全部）
- 逾期任务显示在截止日期单元格

**新口径**：

- 首页采用双层结构：
  - 上层：固定的"历史/逾期区"
    - 逾期区：默认展开，显示未完成且 dueDate < today 的事项
    - 历史区：默认折叠，显示过去日期的已完成事项
  - 下层：主日期流矩阵，从今天开始，固定展示未来 30 天
- Alpha 阶段不做任何筛选功能
- 逾期任务统一显示在逾期区，不在主日期流中占用过去日期行

**修订建议**：

- PRODUCT\_v0.2.md 4.1 节：重写首页布局规格
- PROTOTYPE\_v0.2.md 1.2 节：重写整体结构，移除筛选栏
- TECH\_SPEC\_v0.2.md D.1 节：新增双层结构数据组织方式

***

### 冲突 2：逾期状态存储

**所在文档**：PRODUCT\_v0.2.md、RULES\_v0.2.md、TECH\_SPEC\_v0.2.md

**原口径**：

- Item 状态枚举：pending / completed
- 逾期作为状态之一（部分描述暗示可持久化）

**新口径**：

- 数据库只存两种状态：pending / completed
- overdue = (status == pending) && (dueDate < today)
- 逾期是**纯计算状态**，每次查询时实时计算
- 逾期样式、逾期排序、逾期展示位置全部运行时计算
- **禁止**在数据库中持久化 overdue 字段

**修订建议**：

- PRODUCT\_v0.2.md 5.2 节：明确 status 字段只有 pending/completed
- RULES\_v0.2.md 1.2 节：明确 overdue 为计算状态
- TECH\_SPEC\_v0.2.md C.1 节：Item 表移除任何 overdue 相关字段

**修订建议**：

### 冲突 6：分类删除兜底

**所在文档**：PRODUCT\_v0.2.md、PROTOTYPE\_v0.2.md

**原口径**：

- 删除分类时提示：该分类下的任务将变为"未分类"
- 支持批量转移到其他分类（可选能力）

**新口径**：

- 系统增加内置分类"未分类"
  - fixed id: uncategorized

- 删除任意用户分类后，事项**默认全部迁移到 uncategorized**
- 批量转移到其他分类可以保留为附加能力，但默认兜底必须是 uncategorized

**修订建议**：

- PRODUCT\_v0.2.md 4.2 节：新增"未分类"系统分类说明
- PROTOTYPE\_v0.2.md 3.5 节：更新删除分类提示文案

### 冲突 7：过去日期事项显示

**所在文档**：PRODUCT\_v0.2.md、RULES\_v0.2.md、PROTOTYPE\_v0.2.md

**原口径**：

- 逾期任务显示在截止日期单元格
- 截止前3天规则暗示任务在日期流中占用多行

**新口径**：

- 允许选择过去日期
- 若事项未完成，则进入**逾期区**
- 若事项已完成，则进入**历史区**
- 首页必须能承接这些过去日期事项
- **不得要求过去日期回插主日期流**

**修订建议**：

- PRODUCT\_v0.2.md 4.3 节：明确过去日期事项的归属
- RULES\_v0.2.md 第2章：更新显示规则，明确逾期/历史区逻辑
- PROTOTYPE\_v0.2.md 1.2 节：新增历史/逾期区布局

## 三、冻结后的单一口径规则

### 规则 1：首页双层结构

```
┌─────────────────────────────────────────┐
│              顶部导航栏 (56dp)           │  ← 固定，无筛选栏
├─────────────────────────────────────────┤
│  ┌─────────────────────────────────────┐│
│  │ 逾期区（默认展开）                   ││  ← 固定区域
│  │ - 未完成 && dueDate < today         ││
│  │ - 按逾期天数倒序                     ││
│  ├─────────────────────────────────────┤│
│  │ 历史区（默认折叠）                   ││  ← 固定区域
│  │ - 已完成 && dueDate < today         ││
│  │ - 按完成时间倒序                     ││
│  └─────────────────────────────────────┘│
├─────────────────────────────────────────┤
│                                         │
│         主日期流矩阵（今天起30天）        │  ← 可滚动
│                                         │
├─────────────────────────────────────────┤
│     [ 悬浮添加按钮 ]   [ 回到今天 ]       │  ← 悬浮层
└─────────────────────────────────────────┘
```

### 规则 2：状态定义

| 状态        | 存储位置 | 计算方式                                       |
| --------- | ---- | ------------------------------------------ |
| pending   | 数据库  | 直接存储                                       |
| completed | 数据库  | 直接存储                                       |
| overdue   | 运行时  | `(status == pending) && (dueDate < today)` |

### 规则 3：事项归属判定

```
if (dueDate < today) {
  if (status == completed) {
    归属 = 历史区
  } else {
    归属 = 逾期区
  }
} else {
  归属 = 主日期流（按截止前3天规则显示）
}
```

### 规则 4：Alpha 功能范围

| 功能          | Alpha 状态    |
| ----------- | ----------- |
| 主日期流矩阵      | ✅ 必做        |
| 历史/逾期区      | ✅ 必做        |
| 分类管理（含未分类）  | ✅ 必做        |
| 手动添加事项      | ✅ 必做        |
| 快捷添加 - 文本输入 | ✅ 必做        |
| 快捷添加 - 图片输入 | ✅ 必做（全链路）   |
| AI 解析       | ✅ 必做        |
| 任务完成/逾期状态   | ✅ 必做        |
| XY 轴互换      | ⚠️ Alpha 后段 |
| 时间范围筛选      | ❌ Beta      |
| 分类筛选        | ❌ Beta      |
| 状态筛选        | ❌ Beta      |

### 规则 5：系统内置分类

| 分类 ID         | 名称  | 属性                     |
| ------------- | --- | ---------------------- |
| uncategorized | 未分类 | 系统内置、不可删除、不可重命名、不可修改颜色 |

### 规则 6：过去日期处理

| 场景             | 处理方式      |
| -------------- | --------- |
| 创建过去日期事项（未完成）  | 进入逾期区     |
| 创建过去日期事项（已完成）  | 进入历史区     |
| 修改截止日期到过去（未完成） | 移入逾期区     |
| 修改截止日期到过去（已完成） | 移入历史区     |
| 完成逾期事项         | 从逾期区移入历史区 |
| 取消完成历史事项       | 从历史区移入逾期区 |
