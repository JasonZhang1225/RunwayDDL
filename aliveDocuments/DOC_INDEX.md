## 二、文档职责说明

### 2.1 产品源：PRODUCT.md

**职责**：
- 定义产品愿景、目标用户、核心价值
- 定义功能范围和优先级
- 定义验收标准和发布条件
- 作为产品决策的权威来源

**适用场景**：
- 理解"做什么"
- 确认功能边界
- 验收测试参考

### 2.2 规则源：RULES.md

**职责**：
- 定义数据实体结构
- 定义状态流转逻辑
- 定义边界条件处理规则
- 提供示例场景

**适用场景**：
- 理解"数据如何组织"
- 理解"状态如何变化"
- 处理边界情况

### 2.3 技术源：TECH_SPEC.md

**职责**：
- 定义技术栈选型
- 定义项目结构
- 定义核心模块接口
- 提供代码模式和最佳实践

**适用场景**：
- 理解"如何实现"
- 技术方案设计
- 代码审查参考

### 2.4 执行源：TASK_BREAKDOWN.md

**职责**：
- 分解开发任务
- 定义任务依赖关系
- 规划 Sprint 执行顺序
- 跟踪进度

**适用场景**：
- 任务分配
- 进度跟踪
- 风险识别

### 2.5 设计源：PROTOTYPE.md

**职责**：
- 定义页面布局和组件
- 定义交互流程
- 定义边界状态处理
- 提供验收标准

**适用场景**：
- UI 开发参考
- 交互设计评审
- 测试用例编写

### 2.6 决策源：FREEZE_DECISIONS.md

**职责**：
- 记录已冻结的产品决策
- 定义决策优先级
- 解决文档冲突

**适用场景**：
- 理解"为什么这样设计"
- 解决需求冲突
- 变更影响评估

### 2.7 独立规划：VISUAL_THEME_PLAN.md

**职责**：
- 定义视觉隐喻方向
- 规划渐进式视觉实施
- 分阶段视觉目标

**适用场景**：
- 视觉设计参考
- 主题实施规划

## 四、历史归档目录

`Former/` 目录为历史归档，**不参与当前开发**：

| 子目录 | 内容 | 说明 |
|--------|------|------|
| `Former/history/` | 历史版本文档 | 已被替代的旧版本文档 |
| `Former/patches/` | 补丁文档 | 已合并的 PATCH 文件 |
| `Former/reports/` | 报告文档 | 已完成的合并报告 |

**注意**：
- 历史文档仅供参考，不代表当前决策
- 如需查看历史版本，请参考对应目录
- 不要修改历史文档

## 五、文档使用指南

### 5.1 新成员入职

推荐阅读顺序：
1. [FREEZE_DECISIONS.md](./FREEZE_DECISIONS.md) - 理解核心决策
2. [PRODUCT.md](./PRODUCT.md) - 理解产品目标
3. [RULES.md](./RULES.md) - 理解数据规则
4. [TECH_SPEC.md](./TECH_SPEC.md) - 理解技术方案
5. [TASK_BREAKDOWN.md](./TASK_BREAKDOWN.md) - 了解任务分配

### 5.2 开发任务

1. 从 [TASK_BREAKDOWN.md](./TASK_BREAKDOWN.md) 获取任务
2. 参考 [PRODUCT.md](./PRODUCT.md) 理解需求
3. 参考 [PROTOTYPE.md](./PROTOTYPE.md) 理解 UI 规格
4. 参考 [RULES.md](./RULES.md) 理解数据规则
5. 参考 [TECH_SPEC.md](./TECH_SPEC.md) 理解技术实现

### 5.3 需求变更

1. 检查 [FREEZE_DECISIONS.md](./FREEZE_DECISIONS.md) 确认是否与冻结决策冲突
2. 如有冲突，需先更新冻结决策
3. 更新相关文档
4. 更新任务分解
